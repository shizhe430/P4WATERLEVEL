# ESP32-P4 后续迁移流程

当前流程是正确的：先冻结已经跑通的摄像头基线，再迁移低风险通信和 GPIO，最后合并完整养老饮水业务。不要一开始就把所有模块同时接入，否则问题会互相覆盖。

## 阶段 0：冻结摄像头基线

状态：已完成。

- OV2640 DVP 已出图。
- UART0 921600 原始 JPEG 输出可用。
- 当前引脚和问题记录已写入文档。
- 这个版本作为 Git 初版保存，后续任何模块出问题都能回退对照。

验收标准：

- 串口能连续捕获 `FFD8` 到 `FFD9` 的完整 JPEG。
- 20 秒内能稳定获得多帧可打开图片。

## 阶段 1：P4 与外置 C6 串口链路

目标：让 P4 保持主控，C6 继续负责联网。

当前候选接线：

- P4 TX：GPIO31，接 C6 RX
- P4 RX：GPIO36，接 C6 TX
- 波特率：先沿用 STM32 USART3 的 115200

不要把 C6 接到板上 `UART/TX/RX` 区域。那一路是 UART0 GPIO37/GPIO38，当前用于 USB 转串口和 JPEG 调试输出。

执行顺序：

1. 先烧最小 UART 测试固件，把 GPIO31 配成 TX、GPIO36 配成 RX；GPIO31 配置为 TX 后空闲应为高电平。
2. 确认 GPIO31/GPIO36 没有接屏幕、SPI 扩展或其他外设。
3. P4 侧先写最小串口协议：心跳、设备 ID、事件上报、ACK。
4. C6 侧保持原联网代码，只适配 P4 发来的事件。
5. 用固定测试帧验证丢包、粘包和超时处理。

建议先迁移的消息：

- `HELLO`
- 指纹识别结果
- 出水开始/完成
- 异常停水
- 水位结果

## 阶段 2：安全相关 GPIO 和泵控

目标：先保证不会误出水。

执行顺序：

1. 建立 P4 侧硬件抽象：泵 PWM、泵停止、液位输入、按键输入。
2. 默认上电状态必须是停水。
3. 先不接泵或断开负载，只打印将要执行的泵命令。
4. 确认逻辑正确后再接驱动板。

验收标准：

- 上电、复位、异常、摄像头无帧、串口断开时都保持停水。
- “水位”播报和“异常停水”不能互相打断，必须由状态机统一调度。

## 阶段 3：语音模块

目标：恢复识别和播报，但不影响安全状态。

执行顺序：

1. 按 STM32 的 I2C4 协议迁移寄存器读写。
2. 单独验证地址 `0x34`、识别结果寄存器 `0x64`、播报寄存器 `0x6E`。
3. 加入播报队列或最小互斥：正在播报“水位”时，不允许被低优先级提示打断；真正安全异常可以抢占，但要记录原因。

## 阶段 4：OLED、按键、液位、指纹

执行顺序：

1. OLED 先保留 GPIO bit-bang，减少改动。
2. 按键和液位只做输入去抖和状态打印。
3. 指纹 AS608 单独移植 UART 通信和 WAK 输入。
4. 各模块通过后再接入主状态机。

## 阶段 5：完整养老饮水流程

目标：恢复 ST 端已经验证过的业务逻辑，并贴合养老场景。

流程建议：

1. 待机：OLED 显示设备和状态，C6 保持联网。
2. 用户靠近或按键：进入识别流程。
3. 指纹/联网身份确认：确定用户、杯量偏好和禁忌策略。
4. 摄像头判断杯具和水位。
5. 泵控出水：按水位闭环或时间估算控制。
6. 异常处理：无杯、满杯、低液位、识别失败、串口断开都进入统一停水路径。
7. 事件上报：C6 上传饮水记录和异常记录。

## 每阶段提交规则

每迁移一个模块都单独提交一次，提交信息写清楚：

- 改了哪个模块
- 用了哪些 GPIO
- 单模块验证结果
- 是否影响摄像头基线
