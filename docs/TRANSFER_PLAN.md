# ESP32-P4 工程转移总规划

本文档按当前已经确定的硬件选择重新规划：P4 作为主控，外置 ESP32-C6 保持联网模块，OV2640 DVP 摄像头继续使用已经跑通的接线。目标是在不破坏摄像头基线的前提下，把 STM32H7 工程逐步迁移到 ESP32-P4。

## 当前不可变基线

| 项目 | 决策 |
|---|---|
| 主控 | ESP32-P4 |
| 联网 | 外置 ESP32-C6，和 P4 通过 UART 通信 |
| 摄像头 | OV2640 DVP，保留已验证接线 |
| 摄像头 XCLK | 不接，ATK-MC2640 模块自带有源晶振 |
| 调试/JPEG 串口 | UART0 GPIO37/GPIO38，板上 `UART/TX/RX` 区域和 USB 转串口都在这一路 |
| GPIO0-GPIO15 | 当前不用于 3.3V 外设，VDDPST1 实测不在 3.3V |

## 总引脚映射

### 1. 已验证并固定

| 模块 | 信号 | ESP32-P4 GPIO | 状态 | 说明 |
|---|---|---:|---|---|
| OV2640 | SCL | GPIO32 | 固定 | 摄像头 SCCB |
| OV2640 | SDA | GPIO33 | 固定 | 摄像头 SCCB |
| OV2640 | RESET/RST | GPIO16 | 固定 | 保留 ST 成功时序 |
| OV2640 | PWDN | GPIO17 | 固定 | 保留 ST 成功时序 |
| OV2640 | D0-D5 | GPIO18-GPIO23 | 固定 | DVP 数据 |
| OV2640 | D6 | GPIO26 | 固定 | DVP 数据，不能再当 U1TX |
| OV2640 | D7 | GPIO27 | 固定 | DVP 数据，不能再当 U1RX |
| OV2640 | PCLK | GPIO28 | 固定 | DVP 时钟 |
| OV2640 | VSYNC | GPIO29 | 固定 | DVP 场同步 |
| OV2640 | HREF/DE | GPIO30 | 固定 | DVP 行有效 |
| UART0 | TX | GPIO37 | 固定为调试/JPEG | 不接 C6 |
| UART0 | RX | GPIO38 | 固定为调试/JPEG | 不接 C6 |

### 2. 当前选择后的优先分配

| 模块 | 信号 | ESP32-P4 GPIO | 优先级 | 说明 |
|---|---|---:|---|---|
| ESP32-C6 | P4_TX -> C6_RX | GPIO31 | 第一优先 | GPIO31 当前测得 0V，是因为尚未配置为 TX；配置 UART 后空闲应为高 |
| ESP32-C6 | P4_RX <- C6_TX | GPIO36 | 第一优先 | 已测得空闲约 3.3V |
| 语音模块 | I2C SCL | GPIO46 | 第二优先 | 替代被摄像头占用的 GPIO32 |
| 语音模块 | I2C SDA | GPIO47 | 第二优先 | 替代被摄像头占用的 GPIO33 |
| AS608 指纹 | P4_TX -> AS608_RX | GPIO48 | 第三优先 | UART 可通过 GPIO matrix 映射 |
| AS608 指纹 | P4_RX <- AS608_TX | GPIO49 | 第三优先 | UART 可通过 GPIO matrix 映射 |
| AS608 指纹 | WAK | GPIO50 | 第三优先 | 输入脚，按 ST 端逻辑迁移 |
| 泵控 | PWM/EN | GPIO53 | 第二优先 | 用 LEDC PWM，默认上电低电平停水 |

### 3. 后置或需确认的分配

| 模块 | 推荐 GPIO | 条件 | 说明 |
|---|---|---|---|
| OLED bit-bang | GPIO39-GPIO43 中选 5 个 | 不使用 SD 卡接口 | 这些脚在正点原子表中属于 SD1 连接资源，不是第一周优先项 |
| DS18B20 | GPIO44 | 不使用 SD 卡接口 | 1-Wire 时序单独验证 |
| 水箱液位输入 | GPIO34 | 不使用 MIPI 摄像头接口 | GPIO34 在板表中是 CSI_RST，低速输入可候选，但需实测 |
| 冷水按键 | GPIO45 | 不使用触摸屏复位 | 输入上拉，低电平有效 |
| 氛围灯 1 | GPIO52 | 不使用 LCD_RST | 普通输出，后置 |
| 氛围灯 2 | GPIO54 | 不使用 MIPI CSI_PWDN | 普通输出，后置 |

## 不能再使用的引脚/接口

| 引脚/接口 | 原因 |
|---|---|
| GPIO0-GPIO15 | 当前 VDDPST1 不是 3.3V，已经导致摄像头 SCCB 失败 |
| GPIO26/GPIO27 | 已经是摄像头 D6/D7；虽然开发板标 U1TX/U1RX，但不能再给 C6 |
| GPIO32/GPIO33 | 已经是摄像头 SCCB；不能再接语音 I2C |
| 板上 `UART/TX/RX` 区域 | 对应 UART0 GPIO37/GPIO38，当前用于 USB 转串口和 JPEG 调试输出 |
| GPIO37/GPIO38 | 暂不接 C6，避免和上位机图像流、日志、USB 转串口争用 |

## 一周迁移流程

### 阶段 1：冻结摄像头版本

状态：已完成。

验收标准：

- OV2640 能连续输出 JPEG。
- 上位机能解析完整 `FFD8` 到 `FFD9` 帧。
- `docs/KNOWN_ISSUES.md`、`docs/PIN_MAPPING.md` 和本文件同步。

### 阶段 2：C6 UART 最小链路

目标：先把乐鑫赛道最关键的 P4+C6 架构跑通。

执行：

1. 烧录最小 UART 测试固件，配置 `GPIO31` 为 TX、`GPIO36` 为 RX，波特率先用 115200。
2. 复测 GPIO31：配置成 TX 后，空闲应为高电平。
3. P4 周期发送 `P4_HELLO\r\n`，C6 回 `C6_ACK\r\n`。
4. 加入超时和重发，不接入业务逻辑。

验收：

- P4 和 C6 能互相收发 1000 次固定帧无错。
- 摄像头不受影响。

### 阶段 3：安全输出和泵控

目标：先保证不会误出水。

执行：

1. 新建 P4 硬件抽象层：`pump_stop()`、`pump_set_pwm()`、`tank_level_read()`。
2. 泵默认关闭，上电、复位、异常、摄像头无帧、C6 断联都必须停水。
3. 先断开泵负载，只打印 PWM 指令。
4. 验证状态机后再接泵驱动。

验收：

- 任何异常路径都只会进入停水。
- 不再出现“刚播报水位就被异常停水打断”的无序状态，异常抢占必须有明确原因。

### 阶段 4：语音模块

目标：恢复识别和播报，但受状态机统一调度。

执行：

1. 使用 GPIO46/GPIO47 建立新的 I2C bus。
2. 按 STM32 端协议验证地址 `0x34`、识别结果寄存器 `0x64`、播报寄存器 `0x6E`。
3. 加入播报队列：普通提示不互相打断，安全异常可以抢占但必须记录事件。

验收：

- 单独播报稳定。
- 水位播报、出水中提示、异常停水提示不会无序覆盖。

### 阶段 5：指纹、OLED、液位、按键

执行顺序：

1. AS608：GPIO48/GPIO49 + GPIO50，先只做握手和读系统参数。
2. 液位输入和按键：只打印状态，不直接驱动泵。
3. OLED：如果第一周时间紧，可以后置；如果必须展示，优先确认 GPIO39-GPIO43 未使用 SD 卡后再迁移 bit-bang 驱动。
4. DS18B20 和氛围灯后置。

验收：

- 每个模块单独提交。
- 每个模块通过后再接入主状态机。

### 阶段 6：完整养老饮水业务

整合顺序：

1. 待机：C6 在线，P4 摄像头可用，泵关闭。
2. 用户触发：按键、语音或指纹进入服务流程。
3. 身份确认：指纹或 C6 云端策略返回用户偏好。
4. 杯具/水位识别：摄像头给出水位状态。
5. 出水控制：泵 PWM 按水位、目标量、异常状态闭环控制。
6. 异常处理：无杯、满杯、水箱低液位、摄像头无帧、C6 断联、指纹失败都统一停水。
7. 事件上报：C6 上传饮水记录、异常记录、用户 ID、出水量估算。

## Git 提交节奏

每个阶段单独提交：

1. `camera baseline`
2. `p4 c6 uart link`
3. `pump safety output`
4. `voice i2c driver`
5. `fingerprint driver`
6. `oled and status display`
7. `integrate dispenser workflow`

提交前必须检查：

- 摄像头引脚没有被改动。
- `GPIO0-GPIO15` 没有被新模块使用。
- `docs/KNOWN_ISSUES.md` 记录了新踩坑。
- `docs/PIN_MAPPING.md` 和实际代码一致。
